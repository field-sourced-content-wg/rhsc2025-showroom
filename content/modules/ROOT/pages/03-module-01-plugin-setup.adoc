= モジュール 1: Plugin 設定
:source-highlighter: rouge
:toc: macro
:toclevels: 1

toc::[]

Legacy Java Japan カンパニーのシニア開発者として、あなたはチームを代表して大きな挑戦を任されています。数百の Java EE アプリケーションを Quarkus へ移行するプロジェクトのパイロットを担当することになりました。上司からのメッセージは明確です。「手動での移行は限界だ。MTA と Konveyor AI を使って、まずひとつのアプリで移行を自動化してみてほしい。」

プレッシャーを感じながらも、あなたはその可能性に期待しています。まずは開発環境をセットアップし、Konveyor AI プラグインを IDE に接続することから始めましょう。ここで成功すれば、移行プロセス全体が大きく変わります。

このモジュールでは、OpenShift Dev Spaces で開発環境を構築し、Konveyor AI (Developer Lightspeed) プラグインを MTA サーバーに接続し、分析プロファイルと Generative AI を設定するまでの手順を実践します。

== 学習目標

このモジュールを完了すると、以下のことができるようになります:

* OpenShift Dev Spaces にアクセスし、ワークスペースを起動する
* VS Code 拡張機能として Konveyor AI プラグインをインストールし設定する
* Developer Lightspeed の分析プロファイルを作成し、Generative AI を設定する

== エクササイズ 1: Dev Spaces ワークスペースの起動

Legacy Java Japan カンパニーの移行プロジェクトには、一貫した開発環境が欠かせません。まず OpenShift Dev Spaces にアクセスし、今回のラボで使用するワークスペースを起動します。

. ブラウザで Dev Spaces のコンソールを開きます。以下のリンクをクリックしてください。:
+

https://devspaces.{cluster_domain}[DevSpaces を開く, window=_blank]

. Dev Spaces のログイン画面が表示されたら、次の認証情報でログインします:
+
[source,text,subs="attributes+"]

パスワード: {password}
----

. ログイン後、「Create Workspace」画面が開きます。

. ワークスペース作成画面で、Git リポジトリの URL を入力します。今回のラボ用リポジトリは事前に準備されています:
+
[source,text,subs="attributes+"]
----

----

== エクササイズ 2: Konveyor AI プラグインのインストールと設定

ワークスペースが起動しました。次は、Migration Toolkit for Applications (MTA) の AI 機能を IDE から利用するために、Konveyor AI プラグインをインストールして設定します。このプラグインが Legacy Java Japan カンパニーの移行加速の鍵になります。

=== 前提条件

* エクササイズ 1 で Dev Spaces ワークスペースが起動していること
* MTA サーバーの URL が手元にあること

=== 手順

. コマンドパレットを開きます (`Ctrl+Shift+P` または `F1`):
+
[source,bash,role="execute"]
----
echo "コマンドパレットのショートカット: Ctrl+Shift+P"
----

. コマンドパレットに `MTA: Configure Solution` と入力し、設定コマンドを選択します。

. 設定画面が開いたら、以下の項目の設定をしてください。

.. General Settings 項目の「Enable Hub」を有効化
.. 「Hub URL」に {mta_ui_internal_url}
.. 「Insecure connection」を有効化
.. Authentication 項目の「Enable authentication」を有効化
.. 「Username」に `admin`を指定し、「Password」に {mta_admin_password} を指定する

. 「Save」ボタンをクリックして設定を保存します。

設定例

image::module01-rhdl-configuration.png[Developer lightspeed 設定画面,link=self,window=blank,width=700]


. 「Save」ボタンをクリックしたあと、右下に成功したことを示すメッセージが表示されることを確認してください。失敗した場合は認証情報や接続先の確認をしてください。

image::module01-rhdl-success-configuration.png[Developer lightspeed 設定成功,link=self,window=blank,width=700]
 
=== 確認

プラグインの設定が正常に完了したことを確認します。

. VS Code のステータスバーを確認します。Konveyor アイコンが表示されていれば、プラグインが有効になっています。

. コマンドパレット (`Ctrl+Shift+P`) を開き、`Konveyor` と入力します。MTA 関連のコマンドが複数表示されれば、プラグインが正常にインストールされています:
+
[source,text]
----
MTA: Configure Solution Server Credentials
MTA: Run Analysis
MTA: Open Dtails
...
----

=== よくある問題

**問題**: 設定を保存しても MTA URL が反映されない

→ **解決方法**: VS Code ウィンドウを再読み込みします (`Ctrl+Shift+P` → `Developer: Reload Window`)。

== エクササイズ 3: Developer Lightspeed の設定

Hub への接続ができました。次は、Legacy Java Japan カンパニーのパイロットで実際に使う Developer Lightspeed の準備です。分析プロファイルを作成し、Generative AI（大規模言語モデル）を設定して、コード修正提案を受け取れる状態にします。

=== 前提条件

* エクササイズ 2 で Konveyor AI プラグインの Hub 設定が完了していること
* ワークスペースで Java プロジェクト (`coolstore`) が開いていること
* Generative AI 用の API キーが手元にあること（Showroom のラボ情報、またはファシリテーターから提供）

=== 手順

==== Developer Lightspeed の設定項目を確認する

. アクティビティバー（左側）の MTA アイコンをクリックし、MTA サイドバーを開きます。
+
image::module01-mta-sidebar.png[MTA サイドバー,link=self,window=blank,width=400]
+
「No analysis providers are registered」と表示されている場合は、「Install Extension Pack」をクリックし、Java 向けの言語拡張をインストールしてください。

. コマンドパレット (`Ctrl+Shift+P`) を開き、`MTA: Open Analysis View` と入力して選択します。
+
Analysis View が開くと、初期状態ではプロファイル未選択やモデル接続エラーが表示されます。これはこれから設定する内容です。
+
image::module01-analysis-view-initial.png[MTA Analysis View の初期画面,link=self,window=blank,width=700]

. Analysis View 右上の歯車アイコン（Settings）をクリックし、「Get Ready to Analyze」画面を開きます。設定項目は次のとおりです。
+
[cols="1,2,1"]
|===
|設定項目 |説明 |このラボでの目標

|Select Profile
|分析に使うプロファイルを選択・作成する。プロファイル名は LLM へのコンテキストにも使われる
|プロファイルを新規作成する

|Configure Label Selector
|ソース／ターゲット技術でルールを絞り込む（例: `quarkus`）
|ターゲットを `quarkus` に設定する

|Set Rules
|デフォルトルールやカスタムルールの有効化
|デフォルトルールを有効にする

|Hub Configuration
|MTA Hub への接続（エクササイズ 2 で設定済み）
|Completed であること

|Configure GenAI
|Generative AI（LLM）への接続。`provider-settings.yaml` で API キーやモデルを指定する
|モデルへ接続できること
|===
+
image::module01-get-ready-to-analyze.png[Get Ready to Analyze 画面,link=self,window=blank,width=700]

==== 分析プロファイルを作成する

. 「Get Ready to Analyze」画面の「Select Profile」で「Edit in Profile Manager」をクリックします。または、コマンドパレットで `MTA: Manage Analysis Profile` を実行します。

. 新しいプロファイルを作成し、次の値を設定します。
+
[source,text]
----
Profile Name: coolstore-quarkus
Target Technologies: quarkus
Custom Rules: （未選択）
Use Default Rules: 有効
----
+
[NOTE]
====
ターゲット技術に `quarkus` を指定すると、Quarkus 移行向けのデフォルトルールが適用されます。ソース／ターゲットのどちらか一方は必ず設定してください。
====

. プロファイルを保存し、「Get Ready to Analyze」に戻ります。「Select Profile」「Configure Label Selector」「Set Rules」が設定済み（警告が消えた状態）になっていることを確認します。

==== Generative AI を設定する

. 「Get Ready to Analyze」の「Configure GenAI」で「Configure GenAI Settings」をクリックします。または、コマンドパレットで `MTA: Open the Gen AI model provider configuration file` を実行します。
+
`provider-settings.yaml` が開きます。このファイルで、利用する LLM プロバイダーを選び、API キーとモデル名を指定します。

. 使用するプロバイダーのブロックに `&active` アンカーを付け、API キーとモデルを設定します。このラボでは OpenAI 互換の設定例を使います。
+
[source,yaml,subs="attributes+"]
----
models:
  OpenAI: &active
    environment:
      OPENAI_API_KEY: "{genai_api_key}"
    provider: ChatOpenAI
    args:
      model: {genai_model}
----
+
[IMPORTANT]
====
* アクティブにするプロバイダーは、ブロック名の直後に `&active` を 1 つだけ付けてください。他のプロバイダーにあった `&active` は削除します。
* API キーはソース管理にコミットしないでください。このラボではワークスペース内の拡張機能設定としてのみ保持します。
* OpenShift AI 上のモデルや Azure OpenAI など、別プロバイダーを使う場合は、該当ブロックを `&active` にし、公式ドキュメントの環境変数に合わせて編集してください。
====

. ファイルを保存します (`Ctrl+S`)。

. （任意）Agent Mode を使う場合は、コマンドパレットで `Preferences: Open Settings (UI)` を開き、`Extensions` > `MTA` から次を確認します。
+
[source,text]
----
Gen AI: Enabled     … 有効（デフォルトで有効）
Gen AI: Agent Mode  … このラボではオフのままでも可（後続モジュールで利用）
----
+
[NOTE]
====
Generative AI の設定を変更したあとは、拡張機能の再起動またはウィンドウの再読み込み (`Developer: Reload Window`) が必要になることがあります。
====

=== 確認

Developer Lightspeed の準備が整ったことを確認します。

. Analysis View を再度開き（`MTA: Open Analysis View`）、右上の歯車から「Get Ready to Analyze」を確認します。
+
次の状態になっていれば成功です。
+
[source,text]
----
Select Profile            … 設定済み（coolstore-quarkus が選択されている）
Configure Label Selector  … 設定済み（quarkus）
Set Rules                 … 設定済み（デフォルトルール）
Hub Configuration         … Completed
Configure GenAI           … 接続エラーが解消されている（モデルへ接続できる）
----

. Analysis View 本体で、次を確認します。
+
* 「No active profile selected」の警告が出ていない
* 「Generative AI is enabled」と表示されている
* 「Failed to establish connection to the model」や API キー不足（401）のエラーが出ていない

. コマンドパレットで `MTA: Manage Analysis Profile` を開き、作成した `coolstore-quarkus` プロファイルが一覧に表示されることを確認します。

=== よくある問題

**問題**: 「Failed to establish connection to the model」や API キー不足（401）が表示される

→ **解決方法**: `provider-settings.yaml` で `&active` が付いたブロックの API キーが正しいか確認します。キーを更新したらファイルを保存し、必要に応じて `Developer: Reload Window` でウィンドウを再読み込みします。

**問題**: プロファイルを保存しても「No active profile selected」のまま

→ **解決方法**: Profile Manager で作成したプロファイルを明示的に選択し、「Get Ready to Analyze」を閉じて Analysis View に戻ります。それでも解消しない場合はウィンドウを再読み込みします。

**問題**: 「No analysis providers are registered」と表示される

→ **解決方法**: MTA サイドバーの「Install Extension Pack」から、Java 向けの言語拡張パックをインストールします。インストール後、ワークスペースを再読み込みしてください。

== モジュールまとめ

環境のセットアップと Developer Lightspeed の設定を完了しました。「これは使えそうだ」と感じた瞬間があったなら、それは正しい直感です。

**達成したこと:**

* OpenShift Dev Spaces で一貫した開発環境を構築した
* Konveyor AI (Developer Lightspeed) プラグインをインストールし、MTA Hub に接続した
* 分析プロファイルと Generative AI を設定し、AI 支援分析の準備を整えた

**Legacy Java Japan カンパニーへの価値:**

* **環境の一貫性**: Dev Spaces により、すべての開発者が同じ環境で作業できる
* **AI 支援の準備**: Developer Lightspeed を通じて、移行インサイトとコード修正提案に即座にアクセスできる体制が整った
* **スケーラビリティ**: この設定を他の開発者にも展開できるため、チーム全体の移行速度が上がる

**次のステップ:**

モジュール 2 では、実際の Java EE サンプルアプリケーションに対して MTA の分析を実行し、Quarkus への移行で対処すべき箇所を特定します。セットアップで培った基盤をフル活用する場面です。
