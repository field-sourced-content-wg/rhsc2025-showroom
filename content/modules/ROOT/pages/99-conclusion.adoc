お疲れ様でした。Legacy Java Japan カンパニーのパイロットプロジェクトが完了しました。

4 つのモジュールを通じて、あなたは IDE のセットアップから始め、実際のアプリケーション分析、カスタムルールの作成、そして複雑な Java EE アプリの Quarkus への移行まで、一連のワークフローをすべて自分の手で完成させました。あの上司からのメッセージ「手動での移行は限界だ」に、今のあなたは具体的な答えを持っています。

== 習得したスキル

このワークショップで達成した内容を確認します。

=== モジュール 1: Plugin 設定

* ✅ OpenShift Dev Spaces にアクセスし、ワークスペースを起動する
* ✅ VS Code 拡張機能として Konveyor AI プラグインをインストールし設定する
* ✅ MTA サーバーへの接続を確立し、分析が実行可能な状態であることを確認する

=== モジュール 2: サンプルアプリのマイグレーション

* ✅ Konveyor AI プラグインから MTA 分析を実行し結果を読み解く
* ✅ AI が提案するコード変更を確認し適用する (EJB から CDI、pom.xml 更新など)
* ✅ 移行後の Quarkus アプリを OpenShift にビルドしデプロイし動作確認する

=== モジュール 3: カスタムルールの追加

* ✅ MTA のルールエンジンの仕組みと YAML ルール構造を理解する
* ✅ 組織固有のカスタムルール (社内フレームワーク検出) を YAML で作成する
* ✅ カスタムルールを MTA に登録し分析を再実行して検出結果を確認する

=== モジュール 4: サンプルアプリ 2 のマイグレーション

* ✅ カスタムルールを含む完全な MTA 分析を新しいアプリに実行する
* ✅ 分析結果の優先順位を決定し移行計画を策定する
* ✅ AI 提案と手動修正を組み合わせてコードを移行する (JMS から Reactive Messaging、JSF から REST)
* ✅ 移行アプリを OpenShift にデプロイし包括的な動作確認を行う

== 重要なポイント

このワークショップで体験した 3 つの柱を整理します。

. **分析と可視化**: MTA は Java EE アプリの移行コストと変更箇所を自動で洗い出します。何千行ものコードを手動で確認する必要はありません。分析を先に実行することで、移行の全体像が見えます。

. **AI 支援による加速**: Konveyor AI (Developer Lightspeed) は分析結果と連動して、具体的なコード変更を提案します。`javax.*` のインポート置換のような繰り返し作業は自動化し、JMS や JSF への変換のような判断が必要な作業に集中できます。

. **カスタムルールによる組織知識の蓄積**: 社内フレームワークや独自パターンをルールとして登録することで、MTA が組織固有の問題を検出できるようになります。ルールは一度作れば、すべてのアプリの分析に再利用できます。

== Legacy Java Japan カンパニーへの価値

このワークショップで確立したワークフローは、そのまま本番プロジェクトに持ち込めます。

数百あるアプリケーションに対して、今日体験した流れを繰り返し適用できます。カスタムルールが積み上がるほど分析の精度は上がり、AI 提案の適用も速くなります。手動移行では何ヶ月もかかる作業が、体系的なプロセスとして進められます。

=== 次に挑戦できること

. **カスタムルールの拡充**: このワークショップで作成したルールをベースに、組織内の他のレガシーパターンを追加していきます。YAML ルール構造を理解したので、新しいパターンを追加するだけで分析精度が上がります。

. **CI/CD パイプラインへの統合**: MTA の CLI を使って、コードプッシュのたびに自動で移行分析を実行するパイプラインを組めます。移行の進捗を継続的に把握できるようになります。

. **別の移行ターゲットへの応用**: Quarkus 以外にも、Spring Boot、Thorntail、Jakarta EE への移行に MTA を活用できます。同じワークフローが別のターゲットにも使えます。
