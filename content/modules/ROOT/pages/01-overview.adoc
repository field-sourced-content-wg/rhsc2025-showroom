== あなたの役割とミッション

あなたは *Legacy Java Japan カンパニー* のシニア開発者です。同社は数百の Java EE アプリケーションを長年にわたって運用してきましたが、クラウドネイティブへの移行が急務となっています。

CTO からこう告げられました。「我々は Java EE から Quarkus への移行を加速しなければならない。手動での移行は時間もコストも膨大にかかる。Migration Toolkit for Applications と Konveyor AI を使って、移行プロセスを自動化する方法を評価してほしい。」

*あなたのミッション*：Migration Toolkit for Applications (MTA) と Red Hat Developer Lightspeed を活用して、Java EE アプリケーションの Quarkus への移行を実際に体験し、そのプロセスを評価する。

*目指すゴール*：IDE 上で AI 支援を受けながら移行作業を進め、カスタムルールを組み合わせた自動化された移行ワークフローを確立する。

== このワークショップで達成できること

このワークショップを完了すると、Legacy Java Japan カンパニーの移行プロジェクトに直接役立つ実践的なスキルが身につきます。

* *IDE プラグインの設定方法を理解する* → Developer Lightspeed を IDE に接続し、AI 支援を受けながら移行作業を始められる
* *アプリのマイグレーションに必要なステップを把握する* → Java EE から Quarkus への移行手順を体系的に理解し、繰り返し実施できる
* *カスタムルールの作成方法を習得する* → 組織固有のコーディング規約や依存ライブラリに対応したルールを作成し、MTA に適用できる

*技術的な成果*：AI によるコード変換提案と、カスタムルールによる自動解析を組み合わせた移行ワークフローを習得する。

*ビジネスへの貢献*：数百のアプリケーションを抱える Legacy Java Japan カンパニーにとって、手動移行から AI 支援による移行への転換は、開発チームの作業時間を大幅に削減し、移行品質を均一化する。

== 対象者

このワークショップは次のような方を対象としています。

* Java EE アプリケーションの Quarkus への移行を担当する開発者
* MTA または Konveyor AI の導入を検討しているアーキテクト・テックリード
* AI 支援による開発ツールの実用性を評価したいエンジニア
* クラウドネイティブ移行プロジェクトに関わる、実践的な経験を積みたい方

== Legacy Java Japan カンパニーが直面する課題

*状況*：Legacy Java Japan カンパニーは数百の Java EE アプリケーションを社内で運用しており、現在のインフラコストと開発生産性の両面でクラウドネイティブへの移行が急務です。

*移行の壁*（現場のエンジニアが感じているペインポイント）：

* *作業量の多さ*：Java EE API から Quarkus への置き換えはファイル数が多く、手作業では抜け漏れが発生しやすい → 移行後の品質にばらつきが出る
* *属人化*：移行の知識が特定のシニアエンジニアに集中しており、スケールしない → チーム全体の移行速度が上がらない
* *ルールの管理*：組織固有のライブラリや命名規則に対応したチェックが自動化されていない → レビューコストが高い
* *フィードバックの遅さ*：移行後の問題がビルド・テスト段階になって初めて発覚する → 手戻りコストが大きい

*チャンス*：MTA と Developer Lightspeed を組み合わせると、これらの課題にまとめて対応できます。AI によるコード変換提案、カスタムルールによる自動解析、IDE 上でのリアルタイムフィードバックを一つのワークフローに統合します。

== よくある質問

*「MTA と Konveyor AI は何が違うのですか？」*
→ MTA (Migration Toolkit for Applications) は移行分析と変換を行うツール群です。Konveyor AI（Red Hat Developer Lightspeed とも呼ばれます）は、MTA の分析結果を基に AI がコード変換の提案を行う機能です。Module 1 で両者の関係を詳しく説明します。

*「Java EE から Quarkus への移行はどれくらい複雑ですか？」*
→ アプリケーションの規模と使用している API によって異なります。このワークショップでは代表的なパターンをカバーするサンプルアプリを使い、Module 2 で段階的に体験します。

*「カスタムルールを作るには専門知識が必要ですか？」*
→ MTA のルール記法（XML または YAML）の基本を理解すれば作成できます。Module 3 でゼロから書く手順を丁寧に説明します。

*「このワークショップの環境は本番環境と同じ構成ですか？」*
→ このワークショップでは OpenShift クラスター上に MTA と Red Hat OpenShift Dev Spaces を構成済みの環境を使用します。実際の本番環境への適用方法は各モジュールで補足します。
