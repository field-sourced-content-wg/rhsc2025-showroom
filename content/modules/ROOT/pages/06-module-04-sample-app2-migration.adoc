= モジュール 4: サンプルアプリ 2 のマイグレーション
:source-highlighter: rouge
:toc: macro
:toclevels: 1

toc::[]

ここまでの 3 つのモジュールで、あなたは Legacy Java Japan カンパニーの移行プロジェクトに必要なすべてのピースを手に入れました。IDE プラグインを設定し、最初のアプリを移行し、組織独自のカスタムルールも作りました。

最終モジュールでは、これらのスキルをひとつの流れにまとめます。対象は 2 つ目のサンプルアプリケーションです。EJB、JMS、JSF を含む、より複雑な Java EE アプリです。実際の移行プロジェクトで直面するパターンが凝縮されています。

Module 2 のように手順を追って進めるのではなく、今回は自律的に判断しながら進めます。分析結果を読み解き、優先順位を決め、AI 提案と手動修正を組み合わせて移行を完成させます。これが、Legacy Java Japan カンパニーが目指す移行ワークフローの完成形です。

== 学習目標

このモジュールを完了すると、以下のことができるようになります:

* 複数コンポーネント（EJB、JMS、JSF）を含む複雑な Java EE アプリに対して MTA 分析（カスタムルール込み）を実行する
* 標準ルールとカスタムルール両方の検出結果を解釈し、移行の優先順位を決める
* Konveyor AI の提案を適用しながら、JMS から Reactive Messaging、JSF から RESTful API への変換を手動で仕上げる
* 移行後のアプリケーションを Quarkus でビルドし、OpenShift にデプロイして動作を確認する

== エクササイズ 1: 2 つ目のサンプルアプリの準備と MTA 分析

Legacy Java Japan カンパニーの CTO から、最初の移行成功を受けて追加の指示が届きました。「次は注文管理システムだ。EJB でトランザクション管理、JMS でメッセージキュー、JSF でフロントエンドを組んでいる。Module 3 で作ったカスタムルールも使って、まず全体を把握してほしい。」

Module 2 で経験した流れを思い出しながら、今回はより複雑なアプリケーションを分析します。

=== 前提条件

* Module 1 で設定した Konveyor AI プラグインが有効であること
* Module 3 で作成したカスタムルールが MTA サーバーに登録済みであること
* Dev Spaces のターミナルが開いていること

== エクササイズ 2: 分析結果の解釈と移行計画の策定

分析が完了しました。次は検出結果を読み解き、何を優先して移行するかを判断します。標準ルールの検出結果とカスタムルールの検出結果を区別して見ていきます。実際のプロジェクトでもこの判断力が移行の成否を分けます。

=== 手順

. MTA の Issues エンドポイントから検出された問題の一覧を取得します:
+
[source,bash,role="execute"]
----
curl -s -u {user}:{password} \
  "{mta_url}/hub/applications/$APP_ID/analysis/issues" | python3 -c "
import json, sys
data = json.load(sys.stdin)
issues = data if isinstance(data, list) else data.get('issues', [])
print(f'検出された問題の総数: {len(issues)}')
print()

# カテゴリー別に集計
categories = {}
for issue in issues:
    cat = issue.get('category', 'unknown')
    categories[cat] = categories.get(cat, 0) + 1

print('=== カテゴリー別集計 ===')
for cat, count in sorted(categories.items(), key=lambda x: -x[1]):
    print(f'  {cat}: {count} 件')
"
----

. 重要度（effort）でフィルタリングして、対応が必要な項目を確認します:
+
[source,bash,role="execute"]
----
curl -s -u {user}:{password} \
  "{mta_url}/hub/applications/$APP_ID/analysis/issues" | python3 -c "
import json, sys
data = json.load(sys.stdin)
issues = data if isinstance(data, list) else data.get('issues', [])

print('=== 対応が必要な主要な問題 (effort > 0) ===')
for issue in issues:
    effort = issue.get('effort', 0)
    if effort > 0:
        print(f'[effort:{effort}] {issue.get(\"name\", \"?\")}')
        print(f'  ファイル: {issue.get(\"file\", \"?\")}')
        print(f'  ルールセット: {issue.get(\"ruleSet\", \"?\")}')
        print()
" | head -60
----

. カスタムルールの検出結果を標準ルールと区別して確認します:
+
[source,bash,role="execute"]
----
curl -s -u {user}:{password} \
  "{mta_url}/hub/applications/$APP_ID/analysis/issues" | python3 -c "
import json, sys
data = json.load(sys.stdin)
issues = data if isinstance(data, list) else data.get('issues', [])

standard_issues = []
custom_issues = []

for issue in issues:
    ruleset = issue.get('ruleSet', '')
    if 'legacy-java-japan' in ruleset.lower() or 'custom' in ruleset.lower():
        custom_issues.append(issue)
    else:
        standard_issues.append(issue)

print(f'標準ルール検出数: {len(standard_issues)}')
print(f'カスタムルール検出数: {len(custom_issues)}')
print()

print('=== カスタムルールの検出結果 ===')
for issue in custom_issues:
    print(f'  - {issue.get(\"name\", \"?\")} (ファイル: {issue.get(\"file\", \"?\")})')
"
----

=== ステップ 2: JMS から Reactive Messaging への変換

JMS を使ったメッセージングは Quarkus では SmallRye Reactive Messaging に置き換えます。手動での変換パターンを実践します。

. まず、既存の JMS コードを確認します:
+
[source,bash,role="execute"]
----
find /projects/tackle-order-mgmt -name "*.java" | \
  xargs grep -l "javax.jms\|@MessageDriven\|@JMSConnectionFactory" 2>/dev/null
----

. 検出されたファイルの内容を確認します（例: `OrderMessageProcessor.java`）:
+
[source,bash,role="execute"]
----
# 実際のファイルパスに合わせて変更してください
cat /projects/tackle-order-mgmt/src/main/java/com/example/order/OrderMessageProcessor.java
----

. JMS の `@MessageDriven` EJB を Reactive Messaging の `@Incoming` に変換します。変換後のコードパターンは以下の通りです:
+
[source,java]
----
// 変換前: JMS @MessageDriven EJB
@MessageDriven(activationConfig = {
    @ActivationConfigProperty(propertyName = "destination",
                              propertyValue = "queue/orders"),
    @ActivationConfigProperty(propertyName = "destinationType",
                              propertyValue = "javax.jms.Queue")
})
public class OrderMessageProcessor implements MessageListener {
    @Override
    public void onMessage(Message message) {
        // 注文処理ロジック
    }
}

// 変換後: Quarkus Reactive Messaging
@ApplicationScoped
public class OrderMessageProcessor {

    @Incoming("orders")
    public void processOrder(String orderJson) {
        // 注文処理ロジック (同一のビジネスロジック)
    }
}
----

. `application.properties` に Reactive Messaging のチャンネル設定を追加します:
+
[source,bash,role="execute"]
----
cat >> /projects/tackle-order-mgmt/src/main/resources/application.properties << 'EOF'

# Reactive Messaging チャンネル設定
mp.messaging.incoming.orders.connector=smallrye-in-memory
mp.messaging.incoming.orders.broadcast=true
EOF
----

. `pom.xml` に SmallRye Reactive Messaging の依存関係を追加します:
+
[source,bash,role="execute"]
----
# Quarkus Reactive Messaging 拡張を追加
cd /projects/tackle-order-mgmt
./mvnw quarkus:add-extension -Dextensions="smallrye-reactive-messaging"
----

=== ステップ 3: JSF から RESTful API への変換

JSF の `@ManagedBean` と `@ViewScoped` は Quarkus では JAX-RS (RESTEasy) に置き換えます。

. JSF を使っているファイルを確認します:
+
[source,bash,role="execute"]
----
find /projects/tackle-order-mgmt -name "*.java" | \
  xargs grep -l "javax.faces\|@ManagedBean\|@ViewScoped" 2>/dev/null
----

. JSF ManagedBean を JAX-RS Resource クラスに変換します。変換パターンを確認します:
+
[source,java]
----
// 変換前: JSF ManagedBean
@ManagedBean
@ViewScoped
public class OrderBean implements Serializable {
    private List<Order> orders;

    public List<Order> getOrders() {
        return orderService.findAll();
    }

    public void createOrder(Order order) {
        orderService.create(order);
    }
}

// 変換後: JAX-RS Resource
@Path("/api/orders")
@Produces(MediaType.APPLICATION_JSON)
@Consumes(MediaType.APPLICATION_JSON)
@ApplicationScoped
public class OrderResource {

    @Inject
    OrderService orderService;

    @GET
    public List<Order> getOrders() {
        return orderService.findAll();
    }

    @POST
    public Response createOrder(Order order) {
        orderService.create(order);
        return Response.status(201).build();
    }
}
----

. RESTEasy 拡張が pom.xml に含まれているか確認し、必要に応じて追加します:
+
[source,bash,role="execute"]
----
cd /projects/tackle-order-mgmt

# RESTEasy JAX-RS 拡張を確認・追加
grep -q "quarkus-resteasy-reactive-jackson" pom.xml || \
  ./mvnw quarkus:add-extension -Dextensions="resteasy-reactive-jackson"
----

=== よくある問題

→ **解決方法**: インポートの置換が不完全な可能性があります。`javax.*` が残っていないか確認します:
+
[source,bash,role="execute"]
----
grep -rn "import javax\." --include="*.java" /projects/tackle-order-mgmt/src | \
  grep -v "jms\|faces"
----

**問題**: `oc new-app` で ImageStream が見つからないというエラーが出る

→ **解決方法**: oc コマンドのバージョンと接続クラスターを確認します:
+
[source,bash,role="execute"]
----
oc version
oc whoami --show-server
----

**問題**: ヘルスチェックが `DOWN` を返す

→ **解決方法**: Pod のログで起動エラーを確認します:
+
[source,bash,role="execute"]

oc logs deployment/order-mgmt -n order-mgmt-quarkus-{user} | tail -30
